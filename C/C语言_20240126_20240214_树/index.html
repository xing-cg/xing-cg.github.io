BST

Binary Search Tree，二叉搜索树，也称为二叉排序树。规则：相比于根节点的key值，左子树key小，右子树key大。

插入

在当前节点下插入元素，返回新的root。

为什么要求返回一个新的root呢？

  1. 便于返回新的root。根节点不一定不变，比如之后如果有平衡化的操作。
  2. 便于进行递归，因为树本身就是递归结构，完成insert也可以依赖递归进行，在递归过程中，root是动态变化的，层层递进。

如果遇到插入key相同的情况，将value内容替代。

删除

给定一个root，查找特定的key，删除后，返回0或1（C语言中没有bool类型，用int代替）

需要考虑：

  1. 删除的节点如果有子树怎么办？

首先递归查找key，如果找到了指定key，则有以下几种情况：

  1. 要删除的节点为叶子节点：直接删除
  2. 要删除的节点只有一支子树（左子树或右子树）：子树代替上来即可
  3. 要删除的节点有两支子树：找到和该节点差距最小的节点互换，换位置后，转化为1、2的情况。
    1. 左子树的最右侧节点：左子树中最大值
    2. 右子树的最左侧节点：右子树中最小值

remove也是一种递归，终止条件为当前root为空。

  1. 找到节点为叶子节点时，发现返回值应当为PTreeNode，因为你递归删除子树时，需要反馈给上层一个节点地址信息，以更新上层root的孩子链接。
  2. remove和系统库函数名字冲突，需要改名为remove_node

树的遍历方式

树是一种非线性结构，既然要遍历就得按照一定顺序。

  1. 若按深度优先（DFS），从访问根节点的时机来看，有前序、中序、后序。
    1. 首先访问根节点，第二遍历左子树，第三遍历右子树：是前序
    2. 首先遍历左子树，第二访问根节点，第三遍历右子树：是中序
    3. 首先遍历左子树，第二遍历右子树，第三访问根节点：是后序
  2. 若按广度优先（BFS），则按层次遍历。

B树（2-3-(4)树）

B树派生出好多种形态的树，节点可以有很多，也可以是规定好了的，红黑树就可以从2-3树变种而来，因为红黑树是一种多叉树的二叉树实现。

节点形态

有2型节点、3型节点。

2型节点是节点内部有1个key，可以有两个子节点；3型节点是内部有2个key，可以有三个子节点。

插入节点位置的规则遵循BSTree的规则，即从左往右按从小到大的顺序，左侧最小、右侧最大。

除了以上规则以外，2、3、4型节点要么是叶子节点（没有子节点），要么子节点全占满，没有中间状态。

  1. 假设目前有一个内含6的2型节点
  2. 要插入9，由于子树要么空要么满，所以只能将其与6并列放入节点中，变为一个内含6、9的3型节点。
  3. 继续插入7，会进入一个临时状态，即将其与6、9并列，变为一个内含6、7、9的4型节点。但是紧接着，会分裂为3个2型节点：7作为根节点，左子树为6，右子树为9。
  4. 继续插入5，经过对比，向7的左子树6插入，与6并列，变为一个内含5、6的3型节点。
  5. 继续插入4，经过对比，向7的左子树（5、6）插入，与5、6并列，变为一个内含4、5、6的4型节点。紧接着，又会分裂，但这次不是分裂为3个2型节点了，而是把5上推到父节点的7中，保留4、6。因为如果单纯地分别把4、6下裂后，只有左支的深度单独加1，会导致整个树的不平衡，而上推会保持整个树深度同步。而5上推与7并列后，根节点变为3型节点，需要有三个子树，那么9保留原位，而4、6正好可以各占一个子树的位置。所以，最终结果为：整根为（5、7），子树1为4，子树2为6，子树3为9。

实现

  1. 其中一个比较棘手的问题是，凡是两个数以上的比较，都需要一个机制来对比，哪怕是三个数的对比。
  2. 2-3-4树的数据结构不确定，有中间不稳定的4型节点，所以不能简单地用left、middle、right对比。
  3. 如果B树单个节点中有上千个数据，那么就用数组来存放。但是如果此B树是2-3-4树，则数组就显得比较大动干戈了。
  4. 综上原因，我们想到一个巧妙的方法，就是用颜色来代替某些支。由此产生了：红黑树

红黑树（二叉搜索红黑树）

红黑树其实是2-3-4树的一个变形。

红色代表鲜明，意味着在原先的2-3-4树中是同一节点。

通常规定，把原先2-3树（不存在4型节点）同一节点中的较小数裂在根节点的左侧子树。当然也可以人为规定把较大数放在右侧子树，甚至说同时存在放左、放右的情况（在2-3树中最好不要允许同时放左、放右了，因为如果是2-3-4树也会出现这种情况，容易产生混淆）。所以为了编写程序简单，一般都按前者处理（把原先同一节点中的较小数裂在根节点的左侧子树）。

有效形式

  1. 规定只能左红，不能出现右红。
  2. 规定只能连续一支红线，即一个节点只能由一个红线连接，即在2-3树中，3型节点的情况。即不能出现连续两支红线。

极端形式

二叉树的极端形式就是：一支最长、一支最短：左子树最长情况下是一红线一黑线交替穿插。右子树最短情况下是全黑。这种情况下，左支和右支的高度差最大是2倍。这个高度差是可以接受的，因为红黑树相比于其他树来说插入较简单，比如两边最多差1的AVL树插入是很费劲的，时间复杂度太高。而又能兼顾到查找的速度。

应用

STL中的map（dictionary）、set；

Linux中网络编程中epoll用到的数据结构

节点定义

用直观的符号代替不直观的内容（整数，不能是小数）。通常需要typedef，否则在声明、初始化枚举类时，需要写出全称enum Color。

插入

我们可以默认创造出的新节点都是红色的，因为其总是试图与上层节点结合。

  1. 插入到2-型节点：

    1. 比如向一个内含b的2-型节点插入a，如果该2-型节点没有子树，则欲将之插入到左子树，因为已知b没有子树，则相当于在2-3树中合并节点的情况，在红黑树中则将用红线连接此两个节点。

    2. 比如向一个内含b的2-型节点插入c，如果该2-型节点没有子树，则欲将之插入到右子树，因为已知b没有子树，则相当于在2-3树中合并节点的情况。由于默认创建出来的节点是红的，不符合右边不能有红色的规则，那么就需要左旋操作：即让c上位，b下位至c的左子树，且接管c原先的左子树。插入之前，b是黑色，c是红色，插入之后，还需将b、c的颜色交换。

  2. 插入到3-型节点：

    1. 该3-型节点此时一定有左子树，且左子树一定为红色，右子树不可能是红的。

    2. 向（d、g）节点（没有右子树）插入k：k将要插到g的右子树，但k默认为红色， 需要调整。此时相当于在2-3-4树中合并为过渡4-型节点的情况，之后即将分裂为3个2-型节点。而在红黑树中，自然地省去了分裂的步骤，此时需要把d、k节点的红色反转为黑色（flip color）。但是除了反转d、k的颜色。注意！还需要把g的颜色也反转，因为在2-3-4树中的分裂操作中，被上推的节点企图与上层进行结合。

    3. 向（d、g）节点（没有右子树）插入c：c将要插到d的左子树，但c默认为红色， 出现了左支连续红色，需要调整。此时相当于在2-3-4树中合并为过渡4-型节点的情况，之后即将分裂为3个2-型节点，把d上推，把c、g下裂。在红黑树中，则需要右旋操作：g下位至d的右子树，且接管d原先的右子树。
      颜色问题：调整前的颜色为：g黑、d红、c红，调整后的颜色为：d黑、c红、g红。为什么上移的d变黑了？因为d（newRoot）相当于替代了g（oldRoot）的位置，旧根本来就是黑的，所以d要变黑。

      但是此时又出现了一个问题，此时的状态与“向（d、g）节点（没有右子树）插入k”最后颜色反转前一致，需要把三个节点的颜色均反转！

整体过程：相比于BSTree的插入，前面的流程一致，只是多了插入后的调平的操作。

  1. 终止条件是当时的root为空
  2. 要插入的node的key与已存在的节点的key相等时，认为是更新节点值的操作
  3. 调整
  4. 最后红黑树还要注意一条规则是总根一定是黑色的。

原先的4-型节点只是一个临时、不稳定的中间状态，所以稳态下只可能出现左连续两红，不会存在左红右红。

删除

红黑树的删除可以参考BSTree的过程。

BSTree中，如果节点是叶子节点或只有一支子树则直接删除（删除后让该支子树替代root自身即可（叶子节点的子树为NULL）），如果有两个子树则先转化为未来的状态（选择两支子树中相近大小的节点替代自身），即把将要删除的节点下移至叶子节点或者下移至只有一支子树的状态再删除。
